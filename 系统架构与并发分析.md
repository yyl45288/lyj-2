## 三、并发处理问题分析

### 3.1 当前实现的并发问题

#### 问题1：座位预约竞态条件（超卖问题）

**问题描述**：当多个用户同时请求预约同一个座位的相同时段时，可能导致多个用户都成功预约到该座位。

**问题代码位置**：`api/routes/seats.ts` 第41-50行

```typescript
// 检查座位时段冲突
const seatOverlap = reservations.some(r =>
  r.roomId === roomId
  && r.seatId === seatId
  && (r.status === 'pending' || r.status === 'active')
  && newStart < new Date(r.endTime).getTime()
  && newEnd > new Date(r.startTime).getTime()
)
if (seatOverlap) {
  throwError(ErrorCode.SeatOccupied, '该座位在该时段已被预约')
}

// ... 之后直接添加预约
reservations.push(reservation)
```

**竞态场景**：

| 时间 | 用户A | 用户B | reservations数组 |
|------|-------|-------|------------------|
| T1 | 检查座位S1时段10:00-12:00 | - | 无冲突 |
| T2 | - | 检查座位S1时段10:00-12:00 | 无冲突 |
| T3 | 写入预约记录 | - | [A的预约] |
| T4 | - | 写入预约记录 | [A的预约, B的预约] |

**结果**：同一座位同一时段被两个用户预约，造成超卖。

#### 问题2：签到时的竞态条件

**问题代码位置**：`api/routes/study.ts` 第44-63行

```typescript
if (seat.status === 'occupied') {
  throwError(ErrorCode.SeatOccupied, '该座位正在使用中')
}

// ...

seat.status = 'occupied'
seat.occupiedBy = userId
```

**竞态场景**：用户A和用户B同时对同一座位发起签到，可能都成功。

#### 问题3：用户时段冲突检查的竞态

**问题代码位置**：`api/routes/seats.ts` 第52-60行

```typescript
const userTimeOverlap = reservations.some(r =>
  r.userId === userId
  && (r.status === 'pending' || r.status === 'active')
  && newStart < new Date(r.endTime).getTime()
  && newEnd > new Date(r.startTime).getTime()
)
```

**问题**：同一用户在多个终端同时发起预约时，可能绕过"同一时段只能预约一个座位"的限制。

### 3.2 问题根源分析

当前系统使用**内存数组**作为数据存储，所有操作都是**非原子的**：

1. **检查（read）** 和 **写入（write）** 是两个分离的操作
2. 多个请求可以在检查和写入之间插入执行
3. 缺乏并发控制机制（锁、事务等）

### 3.3 解决方案

#### 方案1：引入互斥锁（Mutex）

**原理**：对共享资源（座位、用户预约列表）加锁，确保同一时间只有一个请求可以修改。

**实现示例**：

```typescript
// api/data.ts
const reservationLocks = new Map<string, Promise<void>>()

export async function acquireLock(key: string): Promise<() => void> {
  // 等待前一个锁释放
  while (reservationLocks.has(key)) {
    await reservationLocks.get(key)!
  }
  
  let resolve!: () => void
  const promise = new Promise<void>(r => resolve = r)
  reservationLocks.set(key, promise)
  
  return () => {
    reservationLocks.delete(key)
    resolve()
  }
}

// api/routes/seats.ts
router.post('/reserve', authMiddleware, asyncHandler(async (req: Request, res: Response): Promise<void> => {
  const { roomId, seatId } = req.body
  
  // 对座位加锁
  const unlock = await acquireLock(`${roomId}_${seatId}`)
  try {
    // ... 原有验证逻辑
    
    reservations.push(reservation)
    
    res.status(200).json({ data: reservation, message: '预约成功' })
  } finally {
    unlock()
  }
}))
```

**适用场景**：单进程部署、中等并发量

**优点**：实现简单，不需要修改存储层

**缺点**：多进程部署时锁不共享，需要分布式锁

---

#### 方案2：数据库事务（推荐用于生产环境）

**原理**：使用数据库的事务机制，在事务内完成检查和写入操作。

**SQL示例**：

```sql
BEGIN TRANSACTION;

-- 检查座位时段是否冲突
SELECT COUNT(*) INTO @conflict_count
FROM reservations
WHERE room_id = ? 
  AND seat_id = ?
  AND status IN ('pending', 'active')
  AND start_time < ? 
  AND end_time > ?;

-- 如果有冲突则回滚
IF @conflict_count > 0 THEN
  ROLLBACK;
  SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '该座位在该时段已被预约';
END IF;

-- 检查用户时段是否冲突
SELECT COUNT(*) INTO @user_conflict_count
FROM reservations
WHERE user_id = ?
  AND status IN ('pending', 'active')
  AND start_time < ? 
  AND end_time > ?;

IF @user_conflict_count > 0 THEN
  ROLLBACK;
  SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '该时间段与您的其他预约冲突';
END IF;

-- 创建预约记录
INSERT INTO reservations (...) VALUES (...);

COMMIT;
```

**适用场景**：生产环境、需要持久化存储

**优点**：数据一致性强，支持多进程部署

**缺点**：需要引入数据库，增加系统复杂度

---

#### 方案3：乐观锁 + 版本号

**原理**：使用版本号或时间戳检测并发冲突，冲突时重试。

**实现示例**：

```typescript
interface SeatWithVersion extends Seat {
  version: number
}

async function reserveSeatOptimistic(
  roomId: string, 
  seatId: string, 
  userId: string,
  startTime: string,
  endTime: string
): Promise<Reservation> {
  let attempts = 0
  const maxAttempts = 5
  
  while (attempts < maxAttempts) {
    // 读取座位当前状态和版本号
    const seat = findSeat(roomId, seatId) as SeatWithVersion
    const originalVersion = seat.version
    
    // 检查冲突
    const conflicts = reservations.filter(r =>
      r.roomId === roomId && r.seatId === seatId
      && r.status === 'pending' || r.status === 'active'
      && startTime < r.endTime && endTime > r.startTime
    )
    
    if (conflicts.length > 0) {
      throw new Error('座位已被预约')
    }
    
    // 尝试更新，检查版本号
    if (seat.version !== originalVersion) {
      attempts++
      continue
    }
    
    // 更新座位状态和版本号
    seat.status = 'reserved'
    seat.version++
    
    // 创建预约
    const reservation = createReservation(...)
    reservations.push(reservation)
    
    return reservation
  }
  
  throw new Error('操作过于频繁，请稍后重试')
}
```

**适用场景**：高并发场景，冲突概率较低

**优点**：性能较好，不需要显式锁

**缺点**：冲突时需要重试，增加客户端复杂度

### 3.4 推荐方案总结

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **互斥锁** | 单进程、开发/测试环境 | 实现简单 | 不支持多进程 |
| **数据库事务** | 生产环境、需要持久化 | 数据一致性强 | 需要数据库支持 |
| **乐观锁** | 高并发、低冲突场景 | 性能好 | 冲突时需重试 |

## 五、总结

### 5.1 架构特点

1. **前后端分离**：前端React + 后端Express，通过REST API通信
2. **模块化设计**：功能按领域划分，职责清晰
3. **内存存储**：当前为演示目的使用内存数组，生产环境需替换为数据库
4. **状态丰富**：座位状态(available/occupied/reserved/mine)、预约状态(pending/active/completed/cancelled)

### 5.2 并发问题核心

当前实现的最大问题是**缺乏并发控制机制**，在高并发场景下会出现数据不一致问题。主要风险点：

1. 座位预约超卖
2. 用户重复预约
3. 签到冲突

### 5.3 改进建议

1. **引入锁机制**：对座位和用户级别的操作加锁
2. **持久化存储**：生产环境使用数据库事务
3. **乐观锁优化**：高并发场景使用版本号检测冲突
4. **WebSocket推送**：实时更新座位状态，提升用户体验
